# Puppy Stardew Server 改进文档

## 1. 目标

这个项目的本质不是实现一个真正的《星露谷物语》专用服务器，而是让一个运行在 Docker 里的真实游戏客户端长期扮演房主。

当前架构是：

```text
Docker 容器
  -> Stardew Valley 客户端
  -> SMAPI
  -> 内置 Mod
  -> Shell 自动化脚本
  -> Web 管理面板
```

所以改进目标不是“重写服务器”，而是降低无人房主方案在实际游玩中的失控概率：

- 玩家能不能随时进服
- 晚上睡觉和跨天会不会卡住
- 面板显示是否可信
- 重启、备份、升级后是否容易恢复
- 存档是否安全
- 管理入口是否不会反过来成为风险点

## 2. 当前最核心的问题

### 2.1 容器在线不等于联机可加入

当前健康检查主要看 `StardewModdingAPI` 进程是否存在。这个只能说明游戏进程活着，不能证明多人联机层已经初始化。

实际会出现：

```text
Docker 显示容器正常
Web 面板显示游戏运行
SMAPI 进程存在
玩家却无法加入
```

根本原因是：自动加载存档不一定等于完整走完游戏内的 `CO-OP -> 选择存档 -> Load` 联机初始化流程。

这会直接影响游戏体验。玩家看到“服务器在线”，但连接失败，会以为网络或客户端坏了；实际上是房主客户端没有把联机房间真正开起来。

### 2.2 自动化有一部分依赖外部按键模拟

现在的 `event-handler.sh` 会监听 SMAPI 日志，然后用 `xdotool` 发按键：

- `F9` 启用 Always On Server
- `Enter` 确认 ReadyCheckDialog
- `Esc` / `Enter` 处理晕倒、菜单、对话框

这种方式能用，但不稳定。它依赖窗口焦点、菜单状态、时机和虚拟显示是否正常。

对游戏的影响是：玩家可能偶发遇到无法跨天、睡觉确认无人处理、服务器模式没有重新打开等问题。

### 2.3 面板很多状态来自日志推断

在线人数、游戏日期、暂停状态、事件次数等状态目前大量依赖 SMAPI 日志解析。

日志不是权威状态，只是旁观记录。日志格式变化、日志轮转、启动阶段日志缺失，都可能让面板显示错误。

实际影响是：面板可能显示 0 人在线，但其实有人；也可能显示运行正常，但玩家进不来。

## 3. 改进优先级

| 优先级 | 改进项 | 直接游戏效果 |
| --- | --- | --- |
| P0 | 重启后自动恢复联机初始化 | 玩家不用等管理员 VNC 手动开 Co-op |
| P0 | 增加 SMAPI 权威状态桥 | 面板能判断“游戏运行”和“可加入” |
| P1 | 将按键自动化迁入 SMAPI Mod | 睡觉、确认、重连更稳定 |
| P1 | 改进健康检查和自动恢复 | 避免“绿灯但进不去”的假在线 |
| P1 | 修复备份下载和备份一致性 | 坏档、误操作、升级前后更安全 |
| P2 | 收紧 Web 面板和 manager 权限 | 降低存档和宿主机被误操作或攻击的风险 |
| P2 | 增加自动化测试和版本一致性 | 后续维护不靠手工试错 |

## 4. P0 改进一：重启后自动恢复联机初始化

### 当前问题

容器重启后，自动加载存档可能只完成“读档”，没有完整初始化多人联机服务器。

玩家感知到的是：

- 服务器看起来在线
- 面板有状态
- 但游戏里无法加入
- 最后需要管理员 VNC 进入画面，手动点 Co-op 载入

### 不改的后果

这个项目会一直是“半自动服务器”。只要发生容器重启、宿主机维护、镜像升级、Mod 调整，就可能需要人工救场。

朋友随时上线这件事并不可靠，尤其是跨时区或玩家零散上线时。

### 做了以后的正向效果

玩家最直接的感受是：服务器重启后仍然能自动恢复到可加入状态。

这会把项目从“能长期跑游戏进程”提升到“更接近真正无人值守的联机房主”。

### 建议实现方向

优先级从高到低：

1. 在 SMAPI Mod 中研究并调用游戏的多人加载流程，尽量复用 Co-op 菜单实际走的初始化路径。
2. 如果现有 `ServerAutoLoad.dll` 无源码，考虑重写一个 `ServerCoopAutoLoad` Mod。
3. 暂时无法完全调用内部流程时，可以做一个状态检测加自动提示：如果读档完成但多人层未初始化，面板明确显示“需要 VNC 重新载入”。

### 验收标准

- 容器重启后，不通过 VNC 手动操作，玩家可以直接加入。
- 面板能明确显示：
  - 游戏进程运行中
  - 存档已加载
  - 多人联机层已初始化
  - 当前可加入
- 如果失败，面板不要显示“健康”，而是显示“游戏运行但联机未就绪”。

## 5. P0 改进二：SMAPI 权威状态桥

### 当前问题

面板和状态上报大量依赖日志：

```text
SMAPI-latest.txt
  -> grep / awk / 正则
  -> status.json
  -> Web 面板
```

这条链路能展示趋势，但不适合作为权威状态。

### 不改的后果

面板会继续有误判风险。玩家问“为什么进不来”时，管理员可能看到面板是绿的，排查方向被带偏。

### 做了以后的正向效果

面板可以准确显示游戏内真实状态，例如：

- 当前是否已进入世界
- 当前是否是主机
- 多人服务器是否已初始化
- 当前在线玩家列表
- 游戏日期和时间
- 当前是否暂停
- 当前是否处于菜单、事件、ready check、睡眠流程
- 最后一次自动睡眠是否成功

这样玩家体验会更清楚：不是一个模糊的“在线/离线”，而是知道到底卡在哪一步。

### 建议实现方向

新增一个 SMAPI Mod 或扩展 `AutoHideHost`，定期写结构化状态文件：

```json
{
  "worldReady": true,
  "isServer": true,
  "multiplayerReady": true,
  "joinable": true,
  "saveName": "Farm_123456",
  "season": "spring",
  "day": 12,
  "timeOfDay": 1840,
  "paused": false,
  "onlinePlayers": [
    { "name": "Alice", "id": "..." }
  ],
  "activeMenu": null,
  "currentEvent": null,
  "lastAutomation": {
    "type": "autoSleep",
    "success": true,
    "at": "2026-07-03T10:00:00Z"
  }
}
```

Web 面板优先读这个状态文件。日志解析作为兜底，而不是主数据源。

### 验收标准

- 面板状态不再主要依赖 grep 日志。
- `status-reporter.sh` 能减少对 SMAPI 文本格式的依赖。
- 游戏内在线人数和面板显示一致。
- 联机不可加入时，面板能给出具体原因。

## 6. P1 改进三：将按键自动化迁入 SMAPI

### 当前问题

`event-handler.sh` 监听日志后模拟按键，这属于外部自动化。

问题是：

- 按键可能发给错误菜单
- 游戏窗口可能没有焦点
- 触发时机可能过早或过晚
- 虚拟显示异常会影响自动化

### 不改的后果

偶发卡住会继续存在，而且很难稳定复现。玩家会遇到：

- 所有人都睡了但没有跨天
- ready check 没确认
- 服务器模式突然掉了
- 自动按键把菜单状态弄乱

### 做了以后的正向效果

把这些行为放进 SMAPI 内部后，可以直接调用游戏对象和方法，不需要模拟人按键。

实际效果是：

- 睡觉跨天更稳定
- ReadyCheck 更稳定
- 菜单处理更可控
- 不依赖 VNC 焦点
- 失败原因更容易记录

### 建议实现方向

优先迁移这些逻辑：

1. ReadyCheck：优先用 `SetLocalReady` 或相关 Team API。
2. 自动睡觉：继续在 Mod 内判断在线玩家是否都在床上，并直接触发房主睡觉。
3. Always On Server 启用：优先通过 ModRegistry 或反射启用，而不是 F9。
4. 菜单处理：只处理确定不会破坏玩家进度的菜单。

保留 `xdotool` 作为最后兜底，但面板应显示“兜底按键已触发”。

### 验收标准

- 常规跨天不需要 `xdotool`。
- ReadyCheck 不需要 `Enter` 模拟。
- Always On Server 启用优先走 API 或反射。
- `event-handler.sh` 只作为兜底，不再是核心路径。

## 7. P1 改进四：健康检查分层

### 当前问题

现在健康检查主要看进程：

```text
pgrep -f StardewModdingAPI
```

但进程存在不代表玩家能进。

### 不改的后果

会继续出现“绿色假在线”。管理员和玩家都会被误导。

### 做了以后的正向效果

面板可以分层展示：

```text
容器状态：运行中
游戏进程：运行中
SMAPI：已加载
存档：已加载
多人联机层：已初始化
玩家加入状态：可加入
```

这样玩家进不来时，不会把问题误判为网络、防火墙或客户端问题。

### 建议实现方向

新增健康等级：

- `container_running`
- `game_process_running`
- `smapi_ready`
- `save_loaded`
- `multiplayer_ready`
- `joinable`
- `automation_healthy`

Docker healthcheck 可以保留进程级检查，但 Web 面板应使用更细的状态。

### 验收标准

- 进程在但不可加入时，面板不能显示完全健康。
- 状态页能给出下一步建议，例如“请 VNC 重新通过 Co-op 载入存档”。

## 8. P1 改进五：备份下载和备份一致性

### 当前问题

备份能力对长期农场很关键。当前需要重点看两件事：

1. 前端下载备份时是否能正确带认证信息。
2. 备份时游戏是否正在写存档。

如果游戏正在保存，直接打包目录可能备到半写入状态。

### 不改的后果

短期不影响种地，但长期风险高：

- 升级 Mod 前不敢放心操作
- 坏档后可能没有可用备份
- 面板里有备份但下载或恢复不顺

### 做了以后的正向效果

玩家和管理员敢放心维护服务器：

- 升级前一键备份
- 出问题能下载回滚
- 备份更可能是可恢复状态

### 建议实现方向

1. 下载备份改为前端 `fetch`，带 `Authorization: Bearer <token>`，拿到 blob 后触发浏览器下载。
2. 或实现一次性短期下载 token，避免长期 JWT 出现在 URL。
3. 备份前通过 SMAPI 状态桥确认当前不在保存中。
4. 如果要更稳，可以先复制存档到临时目录，再压缩临时目录。

### 验收标准

- 备份下载按钮可用。
- 下载不泄露长期 JWT 到 URL。
- 游戏保存中时，备份会等待或提示。
- 恢复测试至少覆盖一次真实存档。

## 9. P2 改进六：收紧管理面板和 manager 权限

### 当前问题

Web 面板有高权限能力：

- 改 Steam 凭证和运行配置
- 上传 Mod
- 重启游戏
- 重建容器
- 读取日志
- 发送 SMAPI 输入

`stardew-manager` 还挂了 Docker socket，权限很高。

### 不改的后果

如果面板暴露公网，风险会很大。对游戏世界的影响可能是：

- 存档被删或覆盖
- 恶意 Mod 被上传
- 容器被反复重启
- Steam 凭证或日志信息泄漏
- 宿主机 Docker 环境受影响

### 做了以后的正向效果

农场更安全。管理员可以放心长期运行，而不是每次开面板都担心被扫到。

### 建议实现方向

1. 文档明确建议面板只走内网、VPN 或反向代理鉴权。
2. `stardew-manager` 只监听 Docker 内部网络。
3. manager 的 `/recreate` 加内部 shared secret。
4. JWT 从 localStorage 迁到 HttpOnly SameSite Cookie。
5. 上传 Mod 前增加 manifest 校验和大小限制提示。

### 验收标准

- 公网部署文档有明确安全警告。
- manager 接口不能被任意容器或外部请求直接调用。
- 面板 token 不容易被前端脚本直接读取。

## 10. P2 改进七：测试和版本维护

### 当前问题

项目里有不少 Shell、Node API、SMAPI Mod 和二进制 DLL。部分 Mod 只有 DLL，没有源码。

这会导致后续维护依赖手工测试。

### 不改的后果

每次更新游戏、SMAPI 或 Mod，都可能破坏自动化链路。问题出现后不容易定位。

### 做了以后的正向效果

维护成本下降。升级前能快速知道哪些能力坏了：

- 登录是否正常
- 配置写入是否安全
- 存档上传是否正确
- 备份是否能创建和下载
- 状态桥是否正常
- Mod 是否加载成功

### 建议实现方向

1. Node API 加单元测试：auth、config、saves、mods、status。
2. Shell 脚本加 smoke test。
3. 给内置 DLL 补源码来源、版本、构建方式。
4. 增加一次完整的部署验收脚本。

### 验收标准

- 改配置、上传存档、创建备份、下载备份都有测试。
- 内置 Mod 版本和源码来源清楚。
- 更新 SMAPI 后能快速跑一遍 smoke test。

## 11. 不建议优先做的方向

### 11.1 不建议尝试重写完整星露谷服务端

星露谷没有官方 dedicated server 协议实现。要重写真正服务端，需要复刻大量游戏逻辑、网络同步和存档行为，成本极高，也容易和客户端版本不兼容。

本项目更现实的方向是继续强化“无人房主”。

### 11.2 不建议继续堆更多按键脚本

更多 `xdotool` 规则只能缓解表面问题，但会让状态机更难维护。

能在 SMAPI 内部解决的，应尽量在 SMAPI 内部解决。

### 11.3 不建议把面板默认暴露公网

面板不是普通展示页，而是管理入口。默认公网暴露会把游戏世界和宿主机控制面扩大。

## 12. 推荐实施路线

### 第一阶段：先让面板状态可信

1. 增加 SMAPI 状态桥。
2. Web 面板优先读取结构化状态。
3. 健康状态拆成进程、存档、多人、可加入四层。

交付效果：玩家进不来时，管理员能马上知道卡在哪。

### 第二阶段：解决重启后自动恢复

1. 研究 Co-op 加载路径。
2. 重写或替换当前 ServerAutoLoad。
3. 确认容器重启后自动进入可加入状态。

交付效果：服务器维护后不需要 VNC 手动救场。

### 第三阶段：减少外部按键模拟

1. ReadyCheck 迁进 SMAPI。
2. Always On Server 启用迁进 SMAPI。
3. 自动睡觉和菜单处理统一由 Mod 负责。
4. `event-handler.sh` 降级为兜底。

交付效果：跨天、睡觉、重连更稳定。

### 第四阶段：补齐运维安全

1. 修复备份下载。
2. 提升备份一致性。
3. manager 加内部鉴权。
4. 面板部署安全文档更新。

交付效果：长期农场更安全，升级和回滚更放心。

## 13. 最小可行改进集

如果只做最少改动，建议先做这四件：

1. 新增 SMAPI 状态桥，输出 `game-state.json`。
2. 面板新增“可加入状态”，不要只显示游戏进程运行。
3. 修复备份下载认证。
4. 面板在检测到“读档完成但联机未就绪”时，明确提示需要 VNC 通过 Co-op 重新载入。

这四项不能完全消灭无人房主的限制，但能显著减少误判和维护成本。

## 14. 最终判断

这个项目值得改，但应该沿着“让无人房主更稳定”这条路改，而不是试图把它变成真正 dedicated server。

最影响实际游戏体验的不是界面，而是：

- 重启后能不能自动恢复可加入
- 睡觉跨天会不会卡
- 面板显示是否可信
- 出事后有没有可恢复备份

优先解决这些，玩家能直接感受到服务器从“能跑”变成“更像一直有人照看的稳定房主”。
